Before you begin contributing, make sure you have the following installed on your system:

- **Node.js** (v18 or higher) - [Download Node.js](https://nodejs.org/)
- **Git** - [Install Git](https://git-scm.com/book/en/v2/Getting-Started-Installing-Git)
- A code editor (we recommend [VS Code](https://code.visualstudio.com/))

## Contributing Process

<Steps>

### Step 1: Fork and Clone the Repository

1. **Fork the repository** on GitHub by clicking the "Fork" button at [AppointyTech/docs](https://github.com/AppointyTech/docs)

2. **Clone your fork** to your local machine:
   ```bash
   git clone https://github.com/[your-username]/docs.git
   cd docs
   ```

3. **Add the upstream remote** to stay synchronized with the main repository:
   ```bash
   git remote add upstream https://github.com/AppointyTech/docs.git
   ```

Before making any changes, create a new branch for your contribution:

```bash

# or
git checkout -b docs/section-you-are-updating
```

Use descriptive branch names like:
- `docs/add-api-authentication-guide`
- `docs/update-saastack-v2-setup`
- `feature/add-component-examples`

### Step 4: Making Changes

#### Creating New Documentation Files

1. **Create a new `.mdx` file** in the appropriate location within the `content/` directory
2. **Add the required frontmatter** at the top of your file:
   ```yaml
   ---
   title: Your Page Title
   description: Optional page description
   ---
   ```

3. **Write your content** using Markdown and MDX syntax

#### Editing Existing Files

1. Navigate to the file you want to edit in the `content/` directory
2. Make your changes using proper Markdown/MDX syntax
3. Save the file and check the live preview in your browser

### Step 5: Testing Your Changes

Before submitting your contribution, thoroughly test your changes:

1. **Check the live preview** in your browser at `http://localhost:3000`
2. **Navigate through your new/updated pages** to ensure proper rendering
3. **Test all links** to make sure they work correctly
4. **Verify responsive design** by testing on different screen sizes
5. **Check for console errors** in your browser's developer tools

#### Build Testing

Test that your changes don't break the build process:

```bash
npm run build
```

If the build succeeds, you're good to proceed. If there are errors, fix them before continuing.

### Step 6: Commit Your Changes

1. **Stage your changes**:
   ```bash
   git add .
   ```

2. **Commit with a descriptive message**:
   ```bash
   git commit -m "docs: add authentication guide for SaaStack v2 API"
   ```

Use conventional commit messages:
- `docs: add new section`
- `docs: update existing content` 
- `docs: fix typos and formatting`
- `feat: add new component`
- `fix: resolve broken links`

### Step 7: Push and Create Pull Request

1. **Push your branch** to your fork:
   ```bash
   git push origin feature/your-feature-name
   ```

2. **Create a Pull Request** on GitHub:
   - Go to your fork on GitHub
   - Click "Compare & pull request"
   - Fill out the PR template with:
     - Clear title describing your changes
     - Detailed description of what you've added/changed
     - Screenshots if you've made visual changes
     - Any breaking changes or special considerations

3. **Request review** from the documentation team

</Steps>

## Documentation Standards

### Writing Guidelines

- **Use clear, concise language** that's accessible to developers of all levels
- **Write in the second person** ("you") for instructional content
- **Use active voice** whenever possible
- **Include code examples** with explanations
- **Break up long content** with headings, lists, and visual elements

### Formatting Standards

- **Use proper heading hierarchy** (H1 â†’ H2 â†’ H3, etc.)
- **Add code syntax highlighting** with appropriate language tags
- **Use callouts** to highlight important information
- **Include step-by-step instructions** for complex procedures
- **Add screenshots** for UI-related documentation (store in `public/images/`)

### Code Examples

Always provide complete, runnable code examples:

```javascript filename="api-call-example.js" copy
// Good: Complete example with error handling
async function fetchUserData(userId) {
  try {
    const response = await fetch(`/api/users/${userId}`, {
      headers: {
        'Authorization': `Bearer ${token}`,
        'Content-Type': 'application/json'
      }
    });
    
    if (!response.ok) {
      throw new Error(`HTTP error! status: ${response.status}`);
    }
    
    return await response.json();
  } catch (error) {
    console.error('Failed to fetch user data:', error);
    throw error;
  }
}
```

### File Organization

- **Group related content** in appropriate directories
- **Use descriptive filenames** with kebab-case (e.g., `api-authentication.mdx`)
- **Create index files** for directory overviews
- **Maintain consistent URL structure** that matches file organization

## Review Process

### What Reviewers Look For

- **Accuracy** - Is the technical information correct?
- **Completeness** - Does it cover all necessary information?
- **Clarity** - Is it easy to understand and follow?
- **Consistency** - Does it match our style and formatting standards?
- **Functionality** - Do all links and code examples work?

### After Review

- **Address feedback promptly** and make requested changes
- **Update your branch** if conflicts arise with the main branch:
  ```bash
  git fetch upstream
  git rebase upstream/main
  ```
- **Request re-review** after making changes

## Getting Help

If you need assistance while contributing:

- **Check existing documentation** for similar examples
- **Review closed PRs** for guidance on structure and style
- **Ask questions in your PR** if you're unsure about something
- **Reach out to the documentation team** for complex technical questions
